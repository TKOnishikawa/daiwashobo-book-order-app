# Architecture Decision Records

## ADR-1: デプロイ先にGitHub Pagesを採用

### 結論
- Next.js静的エクスポート + GitHub Pages + GitHub Actionsでデプロイする

### 判断理由
1. **コスト**: 月額$0。Vercelも無料枠があるが、GitHub Pagesの方がシンプル
2. **既存パイプライン**: dx-projectsで同じ仕組みを運用済み。知見がある
3. **技術的制約なし**: アプリはクライアントサイド完結（SSR不要）。静的エクスポートで十分

### トレードオフ（受容するリスク）
- **公開URL**: 書籍情報は公開データのため機密性リスクは低い。注文書の内容はlocalStorageに保存されURLには含まれない
- **GitHub Pagesの帯域制限**: 月100GBだが、営業部4-5名の利用では問題にならない
- **社内プロキシ**: github.ioがブロックされる可能性 → 動作確認後、必要ならVercelに切替

### 参照
- dx-projectsのデプロイ設定: `.github/workflows/`

## ADR-2: データ永続化にlocalStorageを採用

### 結論
- 注文書データの保存/呼出にlocalStorageを使用する。バックアップとしてJSON DL/UL機能を提供

### 判断理由
1. **サーバーレス**: バックエンド不要でアーキテクチャがシンプル
2. **即座に使える**: ユーザー登録・ログイン不要
3. **オフライン対応**: インターネット切断時も保存データにアクセス可能
4. **十分な容量**: localStorage 5-10MB。注文書データ1件≈1KBなので数千件保存可能

### トレードオフ（受容するリスク）
- **PC間共有不可**: JSON DL/UL機能で手動移動は可能
- **ブラウザクリアで消失**: JSON エクスポートをバックアップ手段として案内
- **複数ブラウザ間で非同期**: 営業担当が1つのブラウザで使う前提

### 参照
- 将来的にkintone連携でクラウド保存に移行する可能性あり（Phase 3）

## ADR-3: Word生成をクライアントサイドで実行

### 結論
- docxライブラリをブラウザ上で実行し、サーバーなしでWord文書を生成する

### 判断理由
1. **サーバー不要**: GitHub Pages（静的ホスティング）と整合
2. **即座にDL**: ネットワーク遅延なし。ブラウザ内で生成→即ダウンロード
3. **既存実装**: word-generator.tsが完成しており、動作確認済み

### トレードオフ（受容するリスク）
- **画像埋め込みの制約**: docxライブラリは画像埋め込み可能だが、CORSの関係で外部URL画像の取得に制約がある → ローカルアップロード画像なら対応可能（Phase 2で検討）
- **フォントの制約**: MS ゴシック指定だが、ユーザー環境にフォントがない場合は代替フォントになる → 営業部PCはWindowsなので問題なし
