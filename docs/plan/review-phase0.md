## 1. プロジェクト概要（1分で読める版）

| 項目 | 内容 |
|------|------|
| **何を作るか** | 大和書房 営業部向け「注文書ジェネレーター」GUIアプリ |
| **誰が使うか** | 営業担当 4-5名 + 長島さん（部長） |
| **現状の課題** | Excel/Word手作業で15-30分/枚。転記ミス・フォーマット属人化 |
| **目標** | ISBN入力→自動補完→Word/PDF出力を3-5分で完了（80%削減） |
| **技術** | Next.js 16（静的エクスポート）→ GitHub Pages。サーバーレス |
| **コスト** | $0（GitHub Pages + クライアントサイド完結） |
| **プロトタイプ** | **80%完成済み**。ISBN検索・フォーム・プレビュー・Word生成が動作 |

### RP-2: GitHub Pages の公開範囲

| 観点 | 内容 |
|------|------|
| **問題** | GitHub Pagesは**パブリックURL**になる |
| **リスク** | 注文書テンプレート自体は機密性低いが、book_master.json（660件の書籍マスタ）が公開される |
| **代替案** | (A) dx-projectsリポ内にサブディレクトリで配置 → 既にpublicリポなので同じ |
|  | (B) 別のprivateリポ + Vercel（無料枠）→ URLはvercel.app |
|  | (C) このままpublicで問題なし（書籍情報は公開データ） |
| **判断** | 書籍マスタの公開がNGかどうか。大和書房側の確認が要るか？ |

### RP-3: リポジトリの配置場所

| 観点 | 内容 |
|------|------|
| **現状** | `dx-projects/clients/daiwashobo/book-order-app/` にプロトタイプが存在 |
| **選択肢** | (A) dx-projects内のサブディレクトリのまま（Astroサイトとは別にActions設定） |
|  | (B) 独立リポ `TKOnishikawa/daiwashobo-book-order-app` に切り出し |
|  | (C) 既存の `TKOnishikawa/daiwashobo-order-automation`（OCR用）に統合 |
| **推奨** | **(B) 独立リポ** — Next.jsのビルド・デプロイがdx-projects（Astro）と干渉しない |
| **判断** | リポの切り出しタイミング（Sprint 0開始前 or 後） |

### RP-4: 保存機能のUX設計

| 観点 | 内容 |
|------|------|
| **問題** | localStorageの保存/呼出UIをどう見せるか |
| **選択肢** | (A) サイドバーに保存リスト常時表示 |
|  | (B) モーダルダイアログで保存/呼出 |
|  | (C) ヘッダーにドロップダウン（Google Docsの「ファイル」メニュー風） |
| **推奨** | **(B) モーダル** — メイン画面のフォーム+プレビューの2カラムを圧迫しない |
| **判断** | Sprint 1で設計確定。Sprint 0のフィードバックも反映 |

### RP-5: Next.js静的エクスポートの技術的制約

| 観点 | 内容 |
|------|------|
| **問題** | `output: 'export'` にすると `next/image` が使えない（`<img>`に置換必要） |
| **現状** | プロトタイプは既に `<img>` を使用（eslint-disableコメント付き） → **問題なし** |
| **確認** | `next.config.js` に `output: 'export'` + `basePath` の設定が必要 |
| **判断** | 技術的にはブロッカーなし。Sprint 0で対応 |

### RP-6: book_master.json の中身と鮮度

| 観点 | 内容 |
|------|------|
| **問題** | 660件のマスタがいつ時点のデータか不明。新刊がカバーされているか |
| **確認** | マスタの最新pub_dateは？大和書房の全書籍数（カタログ）と比較して網羅率は？ |
| **判断** | 現状で十分か、Sprint 0前にマスタ更新が必要か |

### RP-7: MVPの注文書タイプ範囲 ★★ NEW

> 実物サンプル突合で判明した新論点

| 観点 | 内容 |
|------|------|
| **問題** | 実物には**4種類**のフォーマットがある（重版/新刊/新刊予約/フェア） |
| **現状** | プロトタイプはタイプA（重版案内）のみ対応 |
| **選択肢** | (A) **重版案内のみ**で MVP → 最速リリース |
|  | (B) **重版 + 新刊案内**で MVP（推奨） → 主要2タイプをカバー |
|  | (C) 4タイプ全対応 → 工数大。Phase 2以降 |
| **推奨** | **(B) 重版 + 新刊の2タイプ**。新刊は追加フィールド（メ切/配本予定/初版部数）のみの差分で対応可能 |
| **判断** | Sprint 0は重版のみ修正。Sprint 1で新刊フィールド追加。でよいか？ |

### RP-8: SalesRow型の破壊的変更 ★★ NEW

| 観点 | 内容 |
|------|------|
| **問題** | 実物は「書店名 + 冊売」の2列。プロトタイプは「書店名 + 入数 + 売数」の3列 |
| **影響** | SalesRow型から `stock` を削除 → SalesEditor, OrderPreview, word-generator 3ファイルに影響 |
| **判断** | Sprint 0でやるなら早いうちに。既存データとの互換性は気にしなくていい（保存機能未実装のため） |

## 6. 非スコープの明示（「今はやらない」リスト）

以下はMVPに含めない。Phase 2以降で検討。

| 機能 | 除外理由 | 再検討時期 |
|------|---------|-----------|
| 書籍マスタ自動更新（Sheets連携） | MVP不要。手動で十分 | Phase 2 |
| 書影のWord内埋め込み | docxライブラリの制約調査が必要 | Phase 2 |
| 複数書籍の一括Word生成 | 単一書籍で十分な価値提供 | Phase 2 |
| **新刊予約案内（タイプC: 1ページ複数冊）** | レイアウトが根本的に異なる | Phase 2 |
| **フェア案内（タイプD: セット販売）** | テーブル形式で別設計が必要 | Phase 2 |
| **関連書籍「こちらもおススメ！」** | 2冊レイアウトは大改修 | Phase 2 |
| kintone連携 | kintone導入自体が別PJ | Phase 3 |
| POS実績の自動引用 | インテージPJが前提 | Phase 3 |
| ユーザー認証 | 営業部限定利用で不要 | 必要になったら |
| バックエンド/DB | localStorageで十分 | 必要になったら |
